# Urban-Pulse 项目拆解学习资料设计

## 1. 目标

为 Gilbert 编写一套能够从零重新掌握 Urban-Pulse Java 后端的 Markdown 教材集。资料不仅说明现有代码做了什么，还要还原合理的编码顺序、模块依赖、请求链路、Redis/MySQL 数据变化和面试表达，使读者最终能够独立解释项目并承受追问。

## 2. 作者边界

- Gilbert 认领并重点学习 Java 后端：Spring Boot 接口、Redis 登录态、缓存、并发控制、Redis Stream 秒杀下单、MySQL 持久化和测试。
- Vue 2 与 nginx 页面是配套演示客户端，不描述为 Gilbert 独立完成的前端成果。
- 前端内容只讲与后端联调有关的部分：`/api` 代理、token 请求头、接口调用和 401 跳转。
- 文档必须区分“当前代码实际实现”“推荐设计”和“可继续改进”，不能把未实现能力写成项目成果。

## 3. 组织方式

教材按业务模块组织，而不是按目录逐文件罗列。核心链路逐段引用和解释真实代码；Entity、Mapper、Service 接口及重复 CRUD 使用代表文件讲清通用规律，再提供其余文件的对应表。

## 4. 每章固定结构

每个业务模块按以下顺序写作：

1. 本章目标与前置知识。
2. 业务问题：为什么需要这个模块。
3. 从零编码顺序：先定义什么，再实现什么，以及依赖原因。
4. 请求或消息调用链：入口、关键方法、数据存储和返回值。
5. 源码地图：列出真实文件及各自职责。
6. 代码拆解：按完整逻辑块引用源码并逐段说明。
7. 数据变化：Redis Key、数据类型、TTL、MySQL 表和事务边界。
8. 设计取舍：为什么采用当前方案，替代方案有什么差异。
9. 当前问题：指出真实 bug、限制和生产环境缺口。
10. 面试表达：30 秒版本和 2 分钟版本。
11. 高频追问：给出可直接复习的回答。
12. 自测清单：不看答案能够复述后才算学完。

## 5. 核心模块的编码链路

### 5.1 登录鉴权

`User/UserDTO` → `Result` → 验证码接口 → Redis 验证码 → 登录接口 → UUID token → Redis Hash 用户摘要 → `UserHolder` → token 刷新拦截器 → 登录拦截器 → MVC 注册顺序 → 前端携带 token。

### 5.2 店铺缓存

基础数据库查询 → Cache Aside → 空值防穿透 → 更新数据库后删除缓存 → 互斥锁重建 → 逻辑过期与异步重建 → 一致性和热点数据取舍。

### 5.3 笔记、点赞与 Feed

笔记 CRUD → ZSet 点赞 → 点赞状态与 Top 用户 → 关注关系 → 发布时推送粉丝收件箱 → ZSet 时间戳 → 基于 `max` 和 `offset` 的滚动分页。

### 5.4 秒杀下单

优惠券表与库存表 → 普通下单 → 一人一单 → 条件扣库存 → Redis 全局 ID → Lua 原子资格校验 → Redis 预扣库存 → `XADD` → Stream 消费者组 → MySQL 事务落库 → 成功后 `XACK` → pending-list 补偿 → 幂等和数据库兜底。

## 6. 代码拆解标准

- 不机械解释每个 import、getter 或 MyBatis-Plus 空接口。
- 每个包含业务判断、并发语义、缓存策略、事务边界或数据转换的代码块都必须解释。
- 解释变量在进入和离开代码块时的含义，不能只把代码翻译成中文。
- Redis 操作必须说明数据结构、Key 形式、Value、TTL 和复杂度相关影响。
- 数据库操作必须说明命中表、查询条件、索引需求、事务和并发风险。
- 对异步代码必须说明请求线程、消费线程、失败位置和恢复路径。
- 源码存在缺陷时原样说明，不为了面试包装而虚构正确性。

## 7. 面试学习成果

完成资料后，Gilbert 应能够：

- 用 30 秒准确介绍项目定位和个人贡献。
- 画出浏览器、nginx、Spring Boot、Redis、MySQL 的整体数据流。
- 从请求入口完整讲出登录、缓存、Feed 和秒杀链路。
- 解释缓存穿透、击穿、雪崩及项目实际采用的策略。
- 解释 Lua 原子性、Redis Stream 消费者组、ACK 和 pending-list。
- 说明为什么 Redis 预扣库存之后 MySQL 仍需 `stock > 0` 兜底。
- 主动指出固定消费者名、TTL 单位、唯一索引和测试覆盖等不足。
- 明确说明前端是配套演示端，不虚构全栈作者身份。

## 8. 质量与验证

- 所有文件名、类名、方法名、Redis Key、接口路径和表名必须从当前源码核对。
- 每章完成后检查相对链接和代码围栏。
- 教材集完成后扫描 Markdown 是否存在 `TODO`、`TBD`、空章节和未闭合代码围栏。
- 生成源码覆盖索引，确保所有后端生产代码文件至少被教材索引或归入通用模式说明。
- 运行项目现有 Maven 测试作为代码基线验证；若环境依赖导致失败，记录真实原因，不宣称通过。

## 9. 非目标

- 不重写 Vue 前端教程。
- 不把当前任务扩大为后端重构或 bug 修复。
- 不为了增加篇幅逐行解释无业务含义的样板代码。
- 不虚构压测数据、线上部署经验、用户量或生产事故。
- 不把推荐的生产级方案描述成当前仓库已经实现。
