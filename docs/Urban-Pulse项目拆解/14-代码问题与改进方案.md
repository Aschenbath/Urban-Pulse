# 14. 代码问题与改进方案

## 怎么使用这张问题表

面试中主动承认合理限制不是自黑。正确表达结构是：

```text
当前怎么做
→ 在什么条件下有问题
→ 我会怎么改
```

不要只背“项目有 bug”，也不要把推荐修复说成已经实现。

## 高优先级问题

| 问题 | 当前证据 | 影响 | 推荐改进 |
| --- | --- | --- | --- |
| `/blog/**` 全部匿名放行 | MvcConfig | 发布/点赞可能未鉴权并 NPE | 只放行公开 GET 路径 |
| 上传与删除匿名且路径校验不足 | UploadController + MvcConfig | 任意文件、路径穿越、删除风险 | 鉴权、规范化路径、类型/大小白名单、对象存储 |
| CacheClient 互斥锁错误释放 | 抢锁失败仍进入 finally DEL | 误删其他线程锁 | 仅 owner 解锁，取消递归，Redisson |
| 逻辑过期双检错误 | 非空包装 JSON 直接按 R 返回 | 不重建、解析错误、锁等待过期 | 重解析 RedisData、finally 解锁、覆盖测试 |
| 一人一单缺数据库唯一索引 | 订单表只有主键 | 并发/重复消费下可能重复订单 | `(user_id,voucher_id)` unique |
| pending 毒消息无限重试 | 20ms while true | 阻塞所有新订单 | 重试计数、退避、死信 Stream、告警 |

前四项中缓存互斥锁和逻辑过期当前未启用，因此不会影响当前店铺 pass-through 路径，但它们不能被当作可靠完成项。

## 登录与权限问题

### 验证码 TTL 单位错误

```text
LOGIN_CODE_TTL = 120
TimeUnit.HOURS
```

实际 120 小时。推荐常量直接表达单位：

```java
Duration LOGIN_CODE_TTL = Duration.ofMinutes(2);
```

### token TTL 过长且语义不清

`36000 MINUTES` 约 25 天。应根据安全需求明确固定过期、滑动过期和最长会话时间。

### 验证码写日志

开发可以模拟短信，但生产日志不能记录验证码和手机号组合。

### 验证码可重复使用

成功登录后没有删除验证码。应一次性消费并增加发送频率限制、错误次数限制和 IP/手机号风控。

### 登出未实现

后端不删除 token。应根据 authorization 删除 Hash，前端再清本地 token。

### loginInterceptor 遗留依赖

构造器接收 StringRedisTemplate，但当前只读取 UserHolder。应删除无用依赖和旧注释代码。

## 秒杀可靠性问题

### 固定消费者名

所有实例是 `c1`。推荐使用唯一实例 ID，并用 `XAUTOCLAIM` 接管宕机消费者的 pending。

### pending 不在启动时主动扫描

重启后若没有新异常，历史 pending 可能不被处理。应启动先恢复、运行时定期恢复。

### Redis 与 MySQL 无补偿

DB 永久失败时 Redis 已扣库存并记录用户。需要订单状态、重试上限、库存/资格补偿和对账任务。

### 没有秒杀时间校验

beginTime/endTime 只保存不校验。应在 Lua 或可靠活动状态缓存中校验。

### Stream 不裁剪

ACK 不删除消息，Stream 会增长。应设置保留策略、MAXLEN 和监控。

### 没有订单状态查询

接口返回 orderId 时订单可能尚未落库。前端需要查询“排队中/成功/失败”，或由服务推送结果。

### Cluster Key 设计

Lua 内部拼接多个 Key，未使用 KEYS/hash tag，不适合直接迁移 Redis Cluster。应把同一业务 Key 放在同一 slot。

## 缓存问题

### 空值缓存的短暂不一致

数据库刚插入店铺但空值 Key 未过期时仍返回不存在。可在新增时删除空值或接受短 TTL。

### 更新后删除缓存可能失败

需要重试/消息机制；不能声称 MySQL 事务同时保证 Redis。

### 店铺类型 ZSet 永不过期

没有更新入口或 TTL。应在分类变更后删除/重建，或增加版本化缓存。

### 缓存线程池生命周期

静态线程池没有优雅关闭和队列监控。Spring 管理的 Executor 更利于配置与停机。

## 数据库问题

- `tb_follow` 缺组合唯一索引。
- 多个外键只有逻辑关联，可能产生孤儿数据。
- 热门博客按 liked 排序可能需要合适索引，但高频更新 liked 又增加索引维护成本。
- Blog 作者补全存在 N+1。
- 订单/活动状态缺少更完整的业务约束。

## API 和分层问题

- Blog 的业务逻辑写在 Controller，ServiceImpl 为空。
- `Result` 没有泛型和错误码。
- 全局异常可能返回 HTTP 200。
- Controller/字段命名不统一，例如 `loginInterceptor` 首字母小写。
- 多处无用 import、过时注释和错误注释，例如秒杀路径参数被写成用户 ID。
- `/upload/blog/delete` 用 GET 执行删除，不符合 HTTP 语义。

## 依赖和配置漂移

- README JDK 17，pom Java 21。
- MySQL 驱动旧 Maven 坐标。
- Spring Boot 2.7.x 已较旧。
- RedissonConfig 未配置密码。
- Java 24 构建出现 Lombok Unsafe 警告。

升级应单独验证，不要为了“新”一次性升级所有依赖。

## 面试中怎么主动讲不足

可以这样说：

> 这个项目是学习型后端，不是直接宣称生产级。我认为秒杀主链路已经把 Lua、Stream、事务和 ACK 的核心关系串起来，但仍缺数据库组合唯一索引、消费者唯一命名、启动时 pending 恢复和死信处理。另外缓存工具中的互斥锁和逻辑过期是备选练习，当前调用没启用，且实现还有锁释放问题。我能区分哪些是当前成果、哪些是下一步修复。

这种回答展示的是边界意识和技术判断。

## 推荐修复顺序

1. 修正权限放行和上传安全。
2. 修正登录 TTL、验证码一次性和登出。
3. 为订单与关注增加数据库唯一约束。
4. 修正 CacheClient 锁和逻辑过期测试。
5. 完善 pending 恢复、消费者名、重试和死信。
6. 增加秒杀时间与订单状态。
7. 用 Testcontainers 建立集成测试。
8. 最后再补关注、Feed、Geo 等新功能。
