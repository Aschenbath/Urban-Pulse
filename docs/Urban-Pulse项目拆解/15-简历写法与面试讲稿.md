## 30 秒项目介绍

> Urban-Pulse 是一个本地生活点评 Java 后端，使用 Spring Boot、MyBatis-Plus、MySQL 和 Redis，实现了登录、店铺、笔记、优惠券等接口。Redis 主要承担 token 登录态、店铺缓存和秒杀入口。秒杀时 Lua 原子完成库存、一人一单和 Stream 投递，后台消费者再事务落库并在成功后 ACK。前端是配套 Vue/nginx 演示页面，不是我的核心成果。

## 2 分钟项目介绍

> 这个项目的业务场景类似本地生活点评，后端用 Spring Boot 和 MyBatis-Plus，MySQL 保存用户、店铺、笔记、优惠券和订单。登录采用验证码加 Redis token，登录成功后把精简 UserDTO 存入 Redis Hash；第一个拦截器恢复用户到 ThreadLocal 并刷新 TTL，第二个拦截器负责受保护接口鉴权，请求结束清理 ThreadLocal。
>
> 店铺详情采用 Cache Aside，正常数据缓存 30 分钟，不存在的店铺写 2 分钟空值防止缓存穿透，更新店铺时先更新数据库再删除缓存。仓库还保留互斥锁和逻辑过期练习，但当前没有启用，而且我识别到锁释放和双检问题，所以不会把它们说成已上线方案。
>
> 最核心的是秒杀。请求线程先生成 orderId，再执行 Lua，在 Redis 内原子检查库存和一人一单、预扣库存并 XADD 到 Stream，然后立即返回同一个 ID。后台消费者用 XREADGROUP 取消息，在 TransactionTemplate 中检查重复订单、执行 `stock > 0` 的条件扣减并保存订单，事务提交后才 ACK。失败消息保留在 pending-list。当前仍缺唯一索引、唯一消费者名、死信和活动时间校验，这些是我会继续完善的地方。

## 登录模块怎么讲

> 验证码按手机号存 Redis String。登录成功生成随机 UUID token，把 User 转为只含 ID、昵称、头像的 UserDTO，再以 Redis Hash 保存。请求经过两个有顺序的拦截器：第一个尝试恢复用户和续期，第二个只对受保护接口检查 ThreadLocal。这样公开接口上的活跃用户也能续期，同时避免重复查 Redis。请求结束必须 remove ThreadLocal，防止线程复用串号。

### 追问：为什么不用 JWT

> Redis token 每次多一次缓存查询，但便于主动登出、集中续期和权限变化；JWT 减少服务端会话存储，但即时失效和刷新更复杂。这个项目为了练习 Redis 登录态采用前者。

## 缓存模块怎么讲

> 店铺详情先查 `cache:shop:{id}`，正常 JSON 直接返回；空字符串表示数据库确认不存在；Key 不存在才查 MySQL。不存在时写短 TTL 空值，存在时写正常缓存。更新采用先更新数据库、再删缓存的 Cache Aside。当前真正启用的是 pass-through，互斥锁和逻辑过期只是仓库备选，而且存在需要修复的问题。

### 追问：穿透、击穿、雪崩

> 穿透是反复查询不存在的数据，项目用空值缓存；击穿是热点 Key 失效瞬间并发打 DB，可用互斥锁或逻辑过期；雪崩是大量 Key 同时失效或 Redis 不可用，需要随机 TTL、限流、熔断和高可用。当前项目只可靠落地了穿透保护。

## 秒杀模块怎么讲

> 请求线程只做 Redis 资格校验，不直接写 MySQL。Java 先生成 orderId，Lua 检查库存和用户 Set，预扣库存、记录用户并 XADD。Stream 消费线程再事务扣 MySQL 库存和插订单，成功提交后 ACK。DB 成功但 ACK 失败会重复投递，所以消费端先查订单做幂等；最终还应增加组合唯一索引。

### 追问：为什么不会超卖

> Redis 层 Lua 把判断和扣减放在一次原子执行里，正常入口不会把 Redis 库存扣成负数；MySQL 层 UPDATE 带 `stock > 0`，即使 Redis/MySQL 短暂不一致，也不会把数据库库存扣成负数。严格来说还需要订单唯一索引和对账补偿，才能把“一人一单”和最终一致性做完整。

### 追问：为什么不是 exactly-once

> 数据库提交后 ACK 前宕机会让消息再次投递，所以是 at-least-once。通过幂等查询和数据库唯一约束让重复消息不产生重复业务结果，而不是依赖消息系统保证 exactly-once。

### 追问：为什么不用 RabbitMQ

> RabbitMQ 的路由、死信、重试和可靠性生态更专业；Redis Stream 的优势是项目已经依赖 Redis，能用更少组件完成消费者组、ACK 和 pending 的学习闭环。生产选型取决于规模和可靠性要求。

## RedisIdWorker 怎么讲

> 高位使用从固定起点开始的秒级时间差，低 32 位使用 Redis 按业务和日期拆分的 INCR 序列，最后位移和按位或组合成 Long。Redis INCR 保证多实例下序列不重复，ID 也大体按时间递增，更适合作为 MySQL 订单主键。

## 被问“你最难的地方是什么”

> 最难的不是写 XADD，而是确定事务和 ACK 的边界。ACK 必须在 MySQL 事务提交后，否则数据库失败会永久丢单；放在提交后则可能重复消费，所以还要做幂等。这让我真正理解了消息可靠性通常是“至少一次投递 + 业务幂等”，不是一句异步就结束。

## 被问“你踩过什么坑”

选真实且能解释的：

> 一个是订单 ID 最初容易在请求线程和消费线程分别生成，导致前端拿到的 ID 与最终订单不一致，因此改成请求线程只生成一次并传入 Lua。另一个是事务自调用问题，Stream 消费线程不能依赖普通 `@Transactional` 自调用，所以用 TransactionTemplate 显式包住落库逻辑。缓存备选实现中我也识别到抢锁失败仍 finally 解锁的问题，这类并发代码必须靠 owner 语义和测试约束。

## 被问“下一步你会做什么”

> 我会先做可靠性而不是继续堆功能：修正鉴权路径和验证码 TTL，为订单加组合唯一索引，给消费者使用唯一名称并在启动时恢复 pending，增加重试上限和死信 Stream；然后用 Testcontainers 覆盖 Lua、Stream 和 MySQL 事务。最后才考虑补 Feed、Geo 等功能。
