# 07. 探店笔记、当前点赞实现与滚动分页缺口

## 先给结论

当前仓库真正实现的是：

- 发布笔记。
- 查询自己的笔记。
- 按点赞总数分页查询热门笔记。
- 点赞总数单向 `+1`。

当前没有实现：

- Redis ZSet 点赞关系。
- 取消点赞。
- 判断当前用户是否点赞。
- 点赞用户 Top N。
- 笔记详情接口。
- Feed 滚动分页。

所以本章会先拆真实代码，再讲如果继续开发应该按什么顺序补全。

## 数据模型

`tb_blog` 保存：

```text
id, shop_id, user_id, title, images, content,
liked, comments, create_time, update_time
```

`Blog` Entity 还包含三个非数据库字段：

```java
@TableField(exist = false)
private String icon;

@TableField(exist = false)
private String name;

@TableField(exist = false)
private Boolean isLike;
```

它们用于组装返回页面的数据。当前只实际填充 `icon` 和 `name`。

## 发布笔记

```java
@PostMapping
public Result saveBlog(@RequestBody Blog blog) {
    UserDTO user = UserHolder.getUser();
    blog.setUserId(user.getId());
    blogService.save(blog);
    return Result.ok(blog.getId());
}
```

### 逻辑链路

```text
请求 JSON → Blog
→ 从 ThreadLocal 取当前用户
→ 覆盖 userId
→ MyBatis-Plus save
→ 返回自增 blogId
```

Controller 不相信前端传来的 `userId`，而是用登录上下文覆盖，这是正确的权限边界。

### 当前权限问题

`MvcConfig` 排除了 `/blog/**`，导致未登录请求也能进入该方法。此时 `UserHolder.getUser()` 为 null，会产生 NullPointerException，最终被全局异常处理为“服务器异常”。

正确做法是只放行真正公开的查询，例如 `/blog/hot`，发布和“我的笔记”必须鉴权。

## 查询自己的笔记

```java
@GetMapping("/of/me")
public Result queryMyBlog(
        @RequestParam(value = "current", defaultValue = "1")
        Integer current) {

    UserDTO user = UserHolder.getUser();
    if (user == null) {
        return Result.fail("用户未登录");
    }

    Page<Blog> page = blogService.query()
            .eq("user_id", user.getId())
            .page(new Page<>(
                    current,
                    SystemConstants.MAX_PAGE_SIZE
            ));

    return Result.ok(page.getRecords());
}
```

这里在 Controller 内再次检查 user，是在补偿拦截器配置过宽的问题。但更好的位置是统一鉴权，而不是每个方法重复判断。

`MAX_PAGE_SIZE` 当前为 10，因此每页最多返回 10 条。

## 查询热门笔记

```java
Page<Blog> page = blogService.query()
        .orderByDesc("liked")
        .page(new Page<>(current, SystemConstants.MAX_PAGE_SIZE));

List<Blog> records = page.getRecords();
```

它直接按数据库 `liked` 倒序分页。这不是 Redis 热榜，排序压力由 MySQL 承担。

### 为每篇笔记补作者

```java
records.forEach(blog -> {
    Long userId = blog.getUserId();
    User user = userService.getById(userId);
    blog.setName(user.getNickName());
    blog.setIcon(user.getIcon());
});
```

如果一页 10 篇笔记，这段可能产生：

```text
1 次查询博客列表
+ 10 次查询作者
= 11 次 SQL
```

这叫 N+1 查询。推荐批量收集 userId 后一次 `listByIds`，再构建 `Map<Long, User>` 回填。

当前没有判空：如果用户记录被删除，`user.getNickName()` 会 NPE。

## 当前“点赞”到底做了什么

```java
@PutMapping("/like/{id}")
public Result likeBlog(@PathVariable("id") Long id) {
    blogService.update()
            .setSql("liked = liked + 1")
            .eq("id", id)
            .update();
    return Result.ok();
}
```

### 优点

`liked = liked + 1` 在数据库内原子执行，比先查出旧值、Java 加一、再 update 更能避免并发丢失更新。

### 但它不是完整点赞系统

- 同一个用户可以无限调用。
- 没有记录谁点赞。
- 不能取消点赞。
- 未登录也可能调用。
- 没检查 update 是否成功。
- `BLOG_LIKED_KEY` 常量存在但没有被使用。

因此面试只能说“实现了简单点赞计数”，不能说“基于 ZSet 实现点赞排行榜”。

## BlogServiceImpl 为什么是空的

```java
@Service
public class BlogServiceImpl
        extends ServiceImpl<BlogMapper, Blog>
        implements IBlogService {
}
```

当前业务都直接在 Controller 使用 MyBatis-Plus 通用方法，Service 实现没有自定义逻辑。

这能跑，但不是理想分层：点赞、作者补全和发布规则都应该下沉 Service，Controller 只处理 HTTP 参数。

## 评论功能现状

虽然存在：

- `tb_blog_comments`
- `BlogComments`
- `BlogCommentsMapper`
- `IBlogCommentsService`
- `BlogCommentsServiceImpl`
- `BlogCommentsController`

但 Controller 为空，没有任何评论接口。数据模型完成不等于业务完成。

## 如果继续做完整点赞，合理顺序是什么

以下是**推荐补全路径，不是当前实现**。

### 第一步：定义 ZSet

```text
Key:    blog:liked:{blogId}
Member: userId
Score:  点赞时间戳
```

ZSet 同时解决：

- `ZSCORE` 判断用户是否点赞。
- `ZADD` 添加点赞关系。
- `ZREM` 取消点赞。
- `ZRANGE` 查询最早点赞用户。

### 第二步：实现切换逻辑

伪代码：

```java
Double score = redisTemplate.opsForZSet()
        .score(key, userId.toString());

if (score == null) {
    // DB liked + 1
    // ZADD userId timestamp
} else {
    // DB liked - 1
    // ZREM userId
}
```

必须考虑 DB 和 Redis 两步并非同一事务。更可靠做法可以引入事件、重试或定期校准。

### 第三步：查询详情时填 isLike

```text
有登录用户 → ZSCORE
无登录用户 → isLike=false
```

### 第四步：Top 用户

取出前 N 个 userId 后要保持 ZSet 顺序。普通 SQL `WHERE id IN (...)` 不保证返回顺序，需要在 Java Map 中按原 ID 列表重排。

## ScrollResult 与滚动分页

仓库有：

```java
public class ScrollResult {
    private List<?> list;
    private Long minTime;
    private Integer offset;
}
```

但没有方法使用它。它通常用于 Feed ZSet：

- score 是发布时间戳。
- `minTime` 是本页最小 score。
- `offset` 处理多条消息时间戳相同的问题。

这属于下一章的未完成 Feed 设计，不是当前热门笔记分页的实现。

## 面试 30 秒讲法

> 当前笔记模块完成了发布、按用户查询和按数据库点赞数排序的热门列表，点赞只是 SQL 原子自增计数。仓库虽然有 BLOG_LIKED_KEY 和 ScrollResult，但 ZSet 点赞关系、取消点赞和 Feed 滚动分页并未实现，所以我不会把这些说成现有成果。如果继续完善，我会把点赞逻辑下沉 Service，用 ZSet 记录 userId 和点赞时间，并处理 Redis 与数据库计数的一致性。
