# 16. 高频追问与自测题

使用方法：先只看问题并口头回答，再展开答案。不要用“我看懂了”代替主动复述。

## 一、项目全貌

### 1. Urban-Pulse 的核心定位是什么？

**答案：** 本地生活点评场景的 Java 后端学习项目。核心是 Spring Boot、MySQL 与 Redis，重点链路为登录态、店铺缓存和 Redis Stream 秒杀。Vue/nginx 是配套演示端。

### 2. 一次页面请求经过哪些组件？

**答案：** 浏览器 → nginx 8080 → `/api` rewrite → Spring Boot 8081 → 拦截器 → Controller → Service → Redis/MyBatis-Plus → MySQL → Result JSON。

### 3. 项目哪些功能当前没有完成？

**答案：** 真正登出、评论接口、关注接口、Feed、ZSet 点赞、签到、Geo 附近店铺、密码登录、订单状态查询等。

### 4. 为什么不能说这是完整全栈项目？

**答案：** 前端不是本人从零实现，且后端也存在未完成模块。正确定位是 Java 后端与 Redis 实践项目。

## 二、Spring Boot 与分层

### 5. Mapper 是空接口为什么能执行 SQL？

**答案：** Mapper 继承 MyBatis-Plus BaseMapper，运行时 MyBatis 生成代理；启动类通过 MapperScan 扫描接口。

### 6. ServiceImpl 提供了什么？

**答案：** 持有 baseMapper，并提供 save、getById、updateById、query、update 等通用 CRUD 和链式构造能力。

### 7. 为什么 Controller 不应该直接堆业务？

**答案：** Controller 应处理 HTTP 参数和响应，业务规则放 Service 才便于复用、事务代理和测试。当前 BlogController 业务偏多是待改进点。

### 8. Result 有什么优缺点？

**答案：** 优点是统一 success/errorMsg/data；缺点是没有泛型、错误码和明确 HTTP 状态。

### 9. @Transactional 自调用为什么可能失效？

**答案：** Spring 通常通过代理拦截外部调用；同一对象内部直接调用方法可能不经过代理。Stream 代码用 TransactionTemplate 显式事务避免该问题。

## 三、登录鉴权

### 10. 验证码在 Redis 中怎样保存？

**答案：** String，Key 为 `login:code:{phone}`，Value 为六位验证码。当前代码错误使用 120 小时 TTL。

### 11. token 登录态怎样保存？

**答案：** Key 为 `login:token:{uuid}`，类型 Hash，字段为 UserDTO 的 id、nickName、icon，当前 TTL 为 36000 分钟并滑动续期。

### 12. 为什么存 UserDTO 而不是 User？

**答案：** 避免手机号、密码、时间等敏感或无关字段进入 Redis 和接口上下文，减小数据体积和泄露面。

### 13. 为什么需要两个拦截器？

**答案：** 第一个全路径尝试恢复用户并续期，第二个只负责受保护接口鉴权，职责分离且公开接口上的活跃用户也能续期。

### 14. 两个拦截器顺序反过来会怎样？

**答案：** 鉴权时 ThreadLocal 还没有用户，有效 token 也会被误判为未登录。

### 15. 为什么 ThreadLocal 必须 remove？

**答案：** Web 线程池会复用线程，不清理可能造成用户串号和对象引用泄漏。

### 16. 当前登出为什么没实现？

**答案：** Controller 直接返回功能未完成，没有删除 Redis token。前端清 sessionStorage 也不能让服务端 token 立即失效。

### 17. `/blog/**` 全部放行有什么问题？

**答案：** 发布、我的笔记和点赞等写/用户接口也匿名放行，可能 NPE 或越权。应只放行明确公开查询。

## 四、缓存

### 18. 缓存穿透是什么？

**答案：** 请求的数据在数据库不存在，正常缓存无法建立，请求持续打 DB。当前用短 TTL 空字符串处理。

### 19. 如何区分缓存不存在和空值缓存？

**答案：** Redis GET 返回 null 表示 Key 不存在；返回空字符串表示命中人工缓存的“数据库不存在”。

### 20. 缓存击穿是什么？

**答案：** 单个热点 Key 失效瞬间，大量请求并发查 DB。可用互斥锁或逻辑过期。

### 21. 缓存雪崩是什么？

**答案：** 大量 Key 同时失效或 Redis 不可用，流量整体落到 DB。可使用随机 TTL、高可用、限流熔断、多级缓存等。

### 22. 当前店铺查询启用了什么策略？

**答案：** queryWithPassThrough，即普通 JSON 缓存加短 TTL 空值。互斥锁和逻辑过期调用被注释。

### 23. 更新店铺为什么先更新 DB 再删缓存？

**答案：** DB 是真值，删除缓存让下次读取回填；比双写缓存简单并减少无效缓存，但仍有短暂一致性窗口和删除失败问题。

### 24. CacheClient 互斥锁实现的关键 bug 是什么？

**答案：** 抢锁失败后递归重试，但当前栈帧 finally 仍会 DEL，可能删除别人的锁；锁 value 也是固定 1，没有 owner 校验。

### 25. 逻辑过期实现为什么不会正确重建？

**答案：** 获锁后的双检看到包装 JSON 非空就直接返回，并把 RedisData 包装当业务对象解析，还未释放锁；逻辑过期 Key 本来就非空，所以异步任务难以提交。

### 26. 逻辑过期的核心取舍是什么？

**答案：** 允许短暂返回旧数据，换取热点访问低延迟和数据库稳定；不适合要求强一致的场景。

## 五、数据库与领域模型

### 27. Voucher 为什么有 exist=false 字段？

**答案：** stock、beginTime、endTime 不在 tb_voucher，而来自 tb_seckill_voucher 联表或请求输入，不能让 MyBatis-Plus当作本表列。

### 28. 订单 ID 为什么是 IdType.INPUT？

**答案：** orderId 在请求线程由 RedisIdWorker 提前生成，需要作为 Stream 消息和数据库主键，不能由 MySQL 自增。

### 29. 一人一单为什么需要唯一索引？

**答案：** “先查再插”有并发窗口；Redis 也可能出现补偿/重复消息。数据库 `(user_id,voucher_id)` unique 是最终约束。

### 30. 金额为什么以分为单位保存 Long？

**答案：** 避免二进制浮点误差，并方便精确比较和计算。

## 六、RedisIdWorker 与锁

### 31. RedisIdWorker 的公式是什么？

**答案：** $ID=(timestampDelta<<32)|dailySequence$。

### 32. 为什么多实例不会生成相同低位？

**答案：** 所有实例对同一个业务日期 Key 执行 Redis INCR，Redis 原子返回不同序列。

### 33. 当前时间计算有什么问题？

**答案：** LocalDateTime.now 取得本地时间，却按 UTC offset 转 epoch，非 UTC 机器有固定偏移。应使用 Instant.now。

### 34. 为什么分布式锁 value 需要 JVM UUID 加线程 ID？

**答案：** 不同 JVM 的线程 ID 会重复，UUID 区分进程实例，线程 ID 区分同 JVM 线程。

### 35. 为什么释放锁要 Lua？

**答案：** owner 检查和 DEL 必须原子，否则检查后锁可能过期并被别人获得，旧线程会误删新锁。

### 36. SimpleRedisLock 比 Redisson 少什么？

**答案：** 可重入、watchdog 自动续期、等待唤醒、公平/联锁等能力。当前核心业务并未使用 Redisson。

## 七、秒杀 Lua

### 37. Lua 接收哪三个参数？

**答案：** voucherId、userId、orderId。

### 38. Lua 做哪四个关键动作？

**答案：** 判断库存、判断一人一单、Redis 预扣库存/记录用户、XADD Stream。

### 39. Lua 为什么防止 Redis 库存并发超卖？

**答案：** Redis 执行脚本期间不穿插其他命令，库存判断和扣减是一个隔离执行单元。

### 40. Lua 原子是否意味着报错自动回滚？

**答案：** 不意味着。脚本中已执行的写操作不会因后续命令运行时错误自动撤销；原子主要指不被其他命令穿插。

### 41. Redis 一人一单 Set 保存什么？

**答案：** Key `seckill:order:{voucherId}`，member 为通过资格的 userId。它不等同于数据库订单一定成功。

### 42. 当前 Lua 有没有检查活动时间？

**答案：** 没有，只检查库存和重复购买。

## 八、Redis Stream

### 43. Stream 消息 ID 和 orderId 有什么区别？

**答案：** Stream ID 用于消息定位和 ACK，由 Redis 生成；orderId 是业务订单主键，由 RedisIdWorker 生成并写进消息字段。

### 44. 消费者组有什么作用？

**答案：** 在组内分配消息、记录消费者和未确认消息，使多消费者协作并支持 pending 恢复。

### 45. ACK 会删除 Stream 原消息吗？

**答案：** 不会，只把消息从消费者组 pending 状态移除。仍需 XTRIM/删除策略控制 Stream 长度。

### 46. 为什么 ACK 必须在数据库事务后？

**答案：** 先 ACK 后 DB 失败会永久丢单；事务后 ACK 最坏是重复投递，可用幂等恢复。

### 47. DB 成功但 ACK 失败会怎样？

**答案：** 消息保持 pending，之后重复处理；查询到订单已存在后直接 ACK，不再次扣库存。

### 48. 为什么这是 at-least-once？

**答案：** 同一消息可能因 ACK 失败被处理多次，系统保证至少尝试一次，业务靠幂等获得单一结果。

### 49. pending-list 当前有哪些缺口？

**答案：** 只在正常消费异常后扫描；只读当前消费者 pending；固定 c1；毒消息无限 20ms 重试；没有死信和重试计数。

### 50. XAUTOCLAIM 能解决什么？

**答案：** 新消费者可以认领其他消费者长期空闲的 pending 消息，用于实例崩溃恢复。

## 九、MySQL 事务与幂等

### 51. 为什么使用 TransactionTemplate？

**答案：** Stream 消费来自手工线程，且同类内部调用可能绕过 `@Transactional` 代理；TransactionTemplate 显式定义事务提交/回滚边界。

### 52. `stock > 0` 条件更新解决什么？

**答案：** 并发事务在数据库层原子判断并扣减，避免数据库库存被减为负数，也为 Redis/MySQL 不一致提供最终库存兜底。

### 53. 查询已有订单如何实现幂等？

**答案：** 重复消息处理前按 userId 和 voucherId 查询；已有订单说明目标状态已达成，不再扣库存，外层可以 ACK。

### 54. 为什么幂等查询仍不够？

**答案：** 两个并发事务可同时查到 0，最终仍需数据库唯一索引阻止双插入。

### 55. Redis 扣成功、MySQL 永久失败怎么办？

**答案：** 当前会无限 pending，Redis 库存和用户资格已改变。需要重试上限、死信、库存/Set 补偿和对账；当前未实现。

## 十、测试与工程判断

### 56. 当前 6 个测试是否说明覆盖完善？

**答案：** 不是。5 个是占位断言，只有 VoucherOrderServiceImplTest 含核心业务验证。

### 57. 订单测试真正验证什么？

**答案：** 返回给前端的 orderId 与传入 Lua/Stream 的 orderId 是同一个值。

### 58. 怎样测试真实 Lua 和 Stream？

**答案：** 使用 Testcontainers 启动临时 Redis/MySQL，执行脚本、写读 Stream、检查 ACK/pending 和数据库最终状态。

### 59. 如何设计并发库存测试？

**答案：** 库存 N、请求 M，最终断言成功数和订单数不超过 N、stock 不小于 0、每用户最多一单。

### 60. 面试时发现项目有问题应该怎么说？

**答案：** 先说当前实现和适用范围，再指出具体失败条件，最后给可落地改进；不要否定整个项目，也不要假装不存在。
