# 09. 优惠券业务与库存预热

## 本章目标

秒杀链路开始之前，必须先理解优惠券数据如何进入 MySQL 和 Redis。Lua 不会凭空产生库存，它读取的是新增秒杀券时预热的 Key。

## 两种优惠券

`Voucher.type`：

```text
0 → 普通券
1 → 秒杀券
```

通用字段在 `tb_voucher`，秒杀特有字段在 `tb_seckill_voucher`。

## Controller 入口

### 新增普通券

```java
@PostMapping
public Result addVoucher(@RequestBody Voucher voucher) {
    voucherService.save(voucher);
    return Result.ok(voucher.getId());
}
```

只写 `tb_voucher`。

### 新增秒杀券

```java
@PostMapping("seckill")
public Result addSeckillVoucher(@RequestBody Voucher voucher) {
    voucherService.addSeckillVoucher(voucher);
    return Result.ok(voucher.getId());
}
```

完整路径是 `POST /voucher/seckill`。

## 秒杀券新增 Service

```java
@Transactional
public void addSeckillVoucher(Voucher voucher) {
    save(voucher);

    SeckillVoucher seckillVoucher = new SeckillVoucher();
    seckillVoucher.setVoucherId(voucher.getId());
    seckillVoucher.setStock(voucher.getStock());
    seckillVoucher.setBeginTime(voucher.getBeginTime());
    seckillVoucher.setEndTime(voucher.getEndTime());
    seckillVoucherService.save(seckillVoucher);

    stringRedisTemplate.opsForValue().set(
            SECKILL_STOCK_KEY + voucher.getId(),
            voucher.getStock().toString()
    );
}
```

### 第一步：保存通用券

`save(voucher)` 插入 `tb_voucher`，MyBatis-Plus 回填自增 ID。

### 第二步：保存秒杀字段

新建 `SeckillVoucher`，使用刚生成的 voucherId 作为主键，插入库存和时间。

### 第三步：预热 Redis 库存

```text
Key:   seckill:stock:{voucherId}
Type:  String
Value: 库存数量
TTL:   无
```

Lua 后续只读取这个 Key，不会在请求时查询 MySQL 库存。

## 事务能覆盖 Redis 吗

`@Transactional` 只保证两个 MySQL 保存处在同一数据库事务。普通 `StringRedisTemplate` 写入不自动加入 MySQL 事务。

一个失败窗口：

```text
MySQL INSERT 执行
→ Redis SET 成功
→ MySQL 提交阶段失败
→ Redis 仍可能保留不存在券的库存
```

反过来，如果 Redis SET 抛 RuntimeException，Spring 会尝试回滚尚未提交的 MySQL 事务。这仍不等于跨存储原子事务。

生产系统可采用提交后事件、消息队列、重试和对账来保证最终一致性。

## 查询店铺优惠券

Controller：

```java
@GetMapping("/list/{shopId}")
public Result queryVoucherOfShop(@PathVariable Long shopId) {
    return voucherService.queryVoucherOfShop(shopId);
}
```

Service：

```java
List<Voucher> vouchers = getBaseMapper()
        .queryVoucherOfShop(shopId);
return Result.ok(vouchers);
```

XML 使用 LEFT JOIN：

```sql
SELECT v.id, v.shop_id, v.title, v.sub_title,
       v.rules, v.pay_value, v.actual_value, v.type,
       sv.stock, sv.begin_time, sv.end_time
FROM tb_voucher v
LEFT JOIN tb_seckill_voucher sv
       ON v.id = sv.voucher_id
WHERE v.shop_id = #{shopId}
  AND v.status = 1
```

普通券也能返回，只是秒杀字段为 null。

## Redis 库存和 MySQL 库存分别做什么

### Redis 库存

- 高并发入口快速判断。
- Lua 中原子预扣。
- 防止所有请求打进数据库。

### MySQL 库存

- 最终持久化真值。
- 消费线程用 `stock > 0` 条件再次扣减。
- 防止 Redis 与数据库暂时不一致时直接超卖。

## 当前时间校验缺口

虽然 `SeckillVoucher` 有 `beginTime/endTime`，但当前 `seckillVoucher()` 和 `seckill.lua` 都没有检查秒杀是否开始或结束。

只要 Redis 库存 Key 存在且大于 0，用户就能下单。因此不能说项目已实现“秒杀时间窗口校验”。

推荐做法：

- 请求前从可靠缓存读取活动状态；或
- 把开始/结束时间放入 Redis，并在 Lua 内原子校验服务器时间；
- 同时校验券状态。

## Redis 库存没有 TTL 的影响

活动结束后 Key 不会自动消失。需要主动清理、设置到结束时间的 TTL，或由活动状态阻止继续使用。

## 面试讲法

> 优惠券拆成通用券表和秒杀券表。新增秒杀券时先保存通用信息，再保存库存和起止时间，同时把库存预热到 `seckill:stock:{id}`。Lua 秒杀入口只操作 Redis 库存，后台消费者再扣 MySQL 库存作为最终兜底。当前 Redis 写入不属于 MySQL 本地事务，而且时间窗口还没有进入 Lua 校验，这是后续需要完善的一致性和业务校验点。
