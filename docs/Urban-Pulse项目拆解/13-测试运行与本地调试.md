# 13. 测试、运行与本地调试

## 环境真值

当前 `pom.xml` 编译目标是 Java 21。2026-07-13 实际使用 JDK 24.0.1 运行 Maven，由 `--release 21` 产出 Java 21 兼容字节码。

README 仍写 JDK 17，已经和构建配置不一致。运行时优先相信 `pom.xml` 和实际构建输出。

## 所需组件

```text
JDK 21+
Maven 3.8+
MySQL 8
Redis 6+
nginx（展示前端时需要）
```

Redis Stream 消费者组要求 Redis 5+，项目说明使用 Redis 6+。

## 第一步：初始化数据库

```sql
CREATE DATABASE IF NOT EXISTS review_platform
DEFAULT CHARACTER SET utf8mb4;
```

导入：

```powershell
mysql -u root -p review_platform `
  < review-backend/src/main/resources/db/review_platform.sql
```

在 PowerShell 中 `<` 的行为与 cmd/bash 不完全相同；若失败，可使用 MySQL 客户端内的 `source` 命令或通过 cmd 执行。重点是确认 11 张表和样例数据实际导入，而不是只看到数据库创建成功。

## 第二步：环境变量

`.env.example`：

```text
SERVER_PORT=8081
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_DATABASE=review_platform
MYSQL_USERNAME=root
MYSQL_PASSWORD=
REDIS_HOST=127.0.0.1
REDIS_PORT=6380
REDIS_PASSWORD=
```

PowerShell 示例：

```powershell
$env:MYSQL_PASSWORD='your_mysql_password'
$env:REDIS_HOST='127.0.0.1'
$env:REDIS_PORT='6380'
```

环境变量只对当前进程及其子进程有效。新开终端需要重新设置，除非写入用户环境变量或启动脚本。

## 第三步：启动后端

```powershell
Set-Location review-backend
mvn spring-boot:run
```

启动过程中重点观察：

1. MySQL 连接是否成功。
2. Redis 是否可连接。
3. RedissonClient 是否因密码配置失败。
4. `XGROUP CREATE stream.orders g1 $ MKSTREAM` 是否成功或返回正常 BUSYGROUP。
5. 8081 是否被占用。

Stream 初始化失败会阻止 `VoucherOrderServiceImpl` 完成 Bean 初始化，可能导致整个应用启动失败。

## 第四步：启动 nginx

在 `nginx-1.18.0` 目录启动 nginx 后访问：

```text
http://localhost:8080
```

后端接口可以绕过前端直接测试：

```text
http://localhost:8081/shop/1
```

如果直接接口正常、页面失败，优先排查 nginx `/api` rewrite、静态资源路径和 Axios。

## 常见调试顺序

### 页面 502

```text
检查 Spring Boot 是否监听 8081
→ 检查 nginx upstream
→ 检查防火墙/端口
```

### 后端启动报 Redis 连接失败

```text
redis-cli -p 6380 ping
```

预期 `PONG`。如果实例实际在 6379，应修改 `REDIS_PORT`。

### 登录后 /user/me 仍为空

依次检查：

1. 登录响应是否返回 token。
2. sessionStorage 是否保存 token。
3. 请求头是否有 `authorization`。
4. Redis 是否存在 `login:token:{token}` Hash。
5. Hash 字段能否转换为 UserDTO。
6. RefreshTokenInterceptor 是否先执行。

### 秒杀一直库存不足

检查：

```redis
GET seckill:stock:{voucherId}
```

Key 不存在也会被 Lua 当成 0。需要通过新增秒杀券接口或手动一致地初始化 MySQL 和 Redis，不能只改一边。

### 消息进入 Stream 但订单没落库

检查：

```redis
XLEN stream.orders
XINFO GROUPS stream.orders
XPENDING stream.orders g1
```

再看消费者日志中的数据库异常。注意 pending-list 当前可能只在正常消费异常后扫描。

## 五个测试源码分别是什么

### VoucherOrderServiceImplTest：唯一有核心业务断言的单测

它：

- mock RedisIdWorker。
- mock StringRedisTemplate。
- 把用户放入 UserHolder。
- 模拟 Lua 返回成功。
- 断言返回 1001。
- 验证 Lua 参数中的 orderId 也是 1001。

它证明：

> 请求线程只生成一次 ID，返回值和消息参数一致。

它没有证明：

- Lua 命令真的执行。
- Redis 库存正确。
- Stream 消费正常。
- MySQL 事务正确。
- ACK/pending 正确。

### ReviewRedisApplicationTests：占位测试

```java
@Test
void contextLoads() {
    assert true;
}
```

类名像 Spring Context 测试，但没有 `@SpringBootTest`，实际上不启动 Spring。它不能证明应用上下文可加载。

### RedisTest：占位测试

```java
@Test
void integrationTestsAreDisabled() {
    assert true;
}
```

不连接 Redis，只保留测试类结构。

### equalsTest：三个占位测试

三个测试都只有 `assert true`。而 Java `assert` 默认还可能未启用；即使启用，true 也永远通过。它们不验证业务。

### internTest：不是 JUnit 测试

只有 `main` 方法，演示字符串池。Surefire 不会把它计入 6 个测试。

## 为什么“6 tests pass”不能写成“测试完善”

测试数量中 5 个是无业务意义的占位断言，只有 1 个测试包含真实交互验证。因此简历可以写“为 orderId 一致性增加 Mockito 单元测试”，不能写“建立完整自动化测试体系”。

## 推荐测试补全顺序

### 第一优先：纯单元测试

- Lua 返回 null/1/2/未知码。
- UserHolder 缺失时的行为。
- RedisIdWorker 位运算和 Key 格式。
- CacheClient 正常值、空值、未命中。

### 第二优先：Service + mock

- DB 扣库存失败时抛异常。
- 已存在订单时不再次扣库存。
- ACK 只在事务方法成功后调用。

### 第三优先：Testcontainers 集成测试

启动临时 MySQL 和 Redis，验证：

- Lua 真实原子逻辑。
- Stream 消费、ACK、pending。
- 数据库唯一约束。
- 缓存 JSON 和 TTL。

### 第四优先：并发测试

固定库存 N，并发请求 M，最终验证：

```text
成功资格数 ≤ N
MySQL 订单数 ≤ N
MySQL stock ≥ 0
同一用户订单数 ≤ 1
```

## 面试讲法

> 项目当前 Maven clean test 在 Java 21 release 下编译 71 个生产源码并通过 6 个测试，但我不会把数字包装成高覆盖率。其中真正的业务单测验证了秒杀返回的 orderId 与 Lua 投递参数一致，其他多数是为了让默认构建不依赖本地 Redis/MySQL 的占位烟雾测试。后续应使用 Testcontainers 覆盖 Lua、Stream、事务和 pending-list。
