# 03. 数据库设计与领域模型

## 本章目标

这一章先回答三个问题：数据存在哪里、Java 对象如何映射、哪些字段只是返回给前端而不在表里。

## 为什么 User 和 UserInfo 分表

`tb_user` 保存登录频繁访问的核心字段：手机号、密码、昵称、头像。`tb_user_info` 保存城市、简介、粉丝数、生日等低频资料。

这种垂直拆分可以让高频登录查询的行更窄，也把账户身份和个人资料分离。不过当前代码并没有复杂到必须分表，它更像为后续扩展预留的模型。

### User 的主键

```java
@TableName("tb_user")
public class User {
    @TableId(value = "id", type = IdType.AUTO)
    private Long id;
    private String phone;
    private String password;
    private String nickName;
    private String icon;
}
```

- `@TableName` 指定表名。
- `IdType.AUTO` 表示 MySQL 自增主键。
- SQL 给 `phone` 建了唯一索引 `uniqe_key_phone`，它是首次登录并发注册时的数据库兜底。

## Entity 和 DTO 的边界

登录成功后不能把完整 `User` 放进 Redis 或返回前端，因为它含有手机号、密码和内部时间字段。项目使用：

```java
public class UserDTO {
    private Long id;
    private String nickName;
    private String icon;
}
```

这叫数据传输对象。它不是表映射，而是接口和登录上下文需要的最小数据。

### 面试说法

> Entity 对应数据库结构，DTO 对应接口边界。登录态只保存用户 ID、昵称和头像，避免将手机号、密码等敏感或无关字段放入 Redis 和响应体。

## Shop 模型

`Shop` 除了店名、类型、地址、经纬度、价格和评分，还有：

```java
@TableField(exist = false)
private Double distance;
```

`exist = false` 表示 MyBatis-Plus 生成 SQL 时不把 `distance` 当成数据库列。它是一个展示字段，本来可以用于 Geo 附近店铺查询，但当前仓库没有真正实现 Geo 查询。

## Blog 模型

数据库只保存笔记本身和计数，但返回页面时还需要作者昵称、头像和“当前用户是否点赞”：

```java
@TableField(exist = false)
private String icon;

@TableField(exist = false)
private String name;

@TableField(exist = false)
private Boolean isLike;
```

当前热门笔记接口只补充 `icon` 和 `name`，并没有填充 `isLike`。字段存在不代表功能已经完成。

## Voucher 为什么拆成两张表

通用券字段放在 `tb_voucher`：

- 商铺、标题、规则。
- 支付金额、抵扣金额。
- 类型和上架状态。

秒杀专属字段放在 `tb_seckill_voucher`：

- `voucher_id`。
- `stock`。
- `begin_time`、`end_time`。

`voucher_id` 同时是秒杀表主键，与通用券一对一。

`Voucher` 中的秒杀字段使用：

```java
@TableField(exist = false)
private Integer stock;
@TableField(exist = false)
private LocalDateTime beginTime;
@TableField(exist = false)
private LocalDateTime endTime;
```

因为这些列不在 `tb_voucher`，它们来自 XML 联表查询或新增秒杀券请求。

## VoucherOrder 的 ID 为什么是 INPUT

```java
@TableId(value = "id", type = IdType.INPUT)
private Long id;
```

订单 ID 不是 MySQL 自增，而是 Java 的 `RedisIdWorker` 提前生成，所以插入时必须由应用传入。

这很重要：HTTP 接口在 MySQL 真正落库前就要返回 orderId。如果依赖数据库自增，就必须等数据库插入完成，失去异步削峰的意义。

## 缺失的数据库约束

`tb_voucher_order` 当前只有主键：

```sql
PRIMARY KEY (`id`)
```

项目用“先查询是否存在订单”实现一人一单，但并发或重复消费的最终防线应增加：

```sql
UNIQUE KEY uk_user_voucher (user_id, voucher_id)
```

这是**推荐改进**，当前 SQL 没有。面试不能说已经使用唯一索引。

类似地，`tb_follow` 也适合增加 `(user_id, follow_user_id)` 唯一约束，避免重复关注；当前没有。

## 外键为什么没有显式建立

表里有 `shop_id`、`user_id`、`voucher_id`，但 SQL 没有 `FOREIGN KEY`。很多互联网项目选择应用层维护关系，减少级联锁和迁移约束；代价是需要业务和定时校验保证没有孤儿数据。

这里不能强行说这是经过完整生产权衡的架构决策，更准确的说法是：当前脚本只使用逻辑关联，没有数据库外键。

## 金额与评分为什么用整数

- `pay_value`、`actual_value` 单位是分，用整数避免浮点误差。
- `score` 把实际评分乘 10 保存。

原则是涉及金额不要直接使用 `double`。例如 0.1 + 0.2 在二进制浮点中不一定精确等于 0.3。

## ScrollResult 为什么存在但没被使用

```java
public class ScrollResult {
    private List<?> list;
    private Long minTime;
    private Integer offset;
}
```

它明显为 Feed 的滚动分页设计：返回数据、当前最小时间戳和同分值偏移量。但仓库没有 Feed 查询方法，所以它目前只是未完成链路的 DTO。

## 面试两分钟讲法

> 项目主要有用户、店铺、笔记、优惠券和订单几组表。优惠券拆成通用券和秒杀券两张一对一表，因为库存和生效时间只属于秒杀场景。订单主键由 RedisIdWorker 在请求线程提前生成，因此 Entity 使用 INPUT 主键，接口可以先返回 ID，再异步落库。用户登录态使用精简 UserDTO，而不是把含密码和手机号的 User Entity 放进 Redis。当前订单表还缺少 userId、voucherId 唯一索引，这是我会优先补的数据库兜底。
