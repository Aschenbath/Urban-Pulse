# 00. 学习路线与项目边界

## 你现在最需要做什么

先不要逼自己立刻背下 Redis Stream。重新掌握一个项目的正确顺序是：

```text
知道项目解决什么问题
→ 认识数据表和分层
→ 跑通普通请求
→ 理解登录上下文
→ 理解缓存
→ 最后攻克并发秒杀
```

如果跳过前面几层直接背 Lua，面试官把问题换一个说法就容易断线。

## 个人贡献的真实口径

### 可以认领

- Java/Spring Boot 后端结构和接口。
- MyBatis-Plus 数据访问。
- Redis 保存验证码和登录 token。
- 双拦截器与 ThreadLocal 用户上下文。
- 店铺查询缓存、空值缓存以及缓存重建工具。
- Redis 全局 ID。
- Lua 秒杀资格校验。
- Redis Stream 消费者组、ACK 和 pending-list 补偿。
- MySQL 事务扣库存和订单落库。
- 对相关测试、注释和构建配置的维护。

### 不要认领

- 不要说“我独立完成了 Vue 前端”。
- 不要说“这是完整生产级电商系统”。
- 不要虚构 QPS、用户量、线上部署和压测结果。
- 不要把尚未实现的关注、Feed、高级点赞功能说成已完成。
- 不要把推荐的 RabbitMQ、唯一索引、死信队列说成当前已经存在。

## 被问前端时怎么答

> 前端使用配套的 Vue 2 静态页面，通过 nginx 转发 `/api` 请求。我主要负责 Java 后端和 Redis 链路。前端部分我能完成接口联调、token 传递和简单页面修改，但不把它作为自己的前端成果。

这不是减分回答。Java 后端面试真正危险的是冒领后被追问 Vue Router、组件通信、Webpack/Vite 构建而答不上来。

## 被问“是不是独立完成”时怎么答

> 后端核心链路是我学习、实现和维护的，配套页面不是我从零开发的。这个项目主要用于证明我对 Spring Boot、MySQL、Redis 缓存和异步秒杀链路的理解。

## 三遍学习法

### 第一遍：复述结构

只看章节开头、调用链和总结。目标是能够说出每个类属于哪一层。

### 第二遍：手动画链路

至少画四张图：

1. 普通 HTTP 请求链路。
2. token 登录链路。
3. 店铺缓存查询链路。
4. 秒杀请求线程与消费线程链路。

### 第三遍：主动回忆

关闭源码，回答每章末尾的自测题。答不出来再回到对应代码块，不要从头重复看全文。

## 面试优先级

| 优先级 | 模块 | 原因 |
| --- | --- | --- |
| S | Redis Stream 秒杀 | 项目最强的技术叙事 |
| S | 店铺缓存 | Redis 常见面试问题密集 |
| A | 登录与拦截器 | 能展示 Web 请求上下文和 Redis Hash |
| A | RedisIdWorker、锁 | 能承接分布式基础问题 |
| B | 数据库与分层 | 所有 Java 面试的基本功 |
| C | 普通博客 CRUD、上传 | 用于补全业务，不应抢占核心篇幅 |
| 未完成 | 关注、Feed、高级点赞 | 只能讲现状和后续设计，不能认领 |

## 焦虑时的最低复习闭环

如果面试就在眼前，先确保能完整说出下面这段：

> Urban-Pulse 是一个本地生活点评后端。我使用 Spring Boot 和 MyBatis-Plus 实现用户、店铺、笔记和优惠券接口，Redis 负责登录态、缓存和秒杀入口。秒杀时 Java 先生成订单 ID，Lua 原子完成库存、一人一单和 Stream 投递，后台消费者在 MySQL 事务中扣库存并保存订单，成功后 ACK，失败消息留在 pending-list 重试。前端是配套演示页面，不是我的主要成果。
