# 02. 从零搭建项目的编码顺序

## 为什么要学“先写什么”

只记住最终代码，会产生一种错觉：每个类都同时存在，不知道为什么需要它。面试官一问“如果让你重写，你从哪里开始”，就容易失去顺序。

下面是更合理的演进路线。它不一定等同于真实 Git 提交顺序，但符合代码依赖关系。

## 阶段一：建立能启动的 Spring Boot 工程

先写：

1. `pom.xml`
2. `ReviewRedisApplication`
3. `application.yaml`
4. 数据库初始化 SQL

此时目标只是让应用能编译、能连接 MySQL/Redis，不急着写复杂业务。

## 阶段二：建立最小分层

以 Shop 为例：

```text
tb_shop
→ Shop entity
→ ShopMapper extends BaseMapper<Shop>
→ IShopService extends IService<Shop>
→ ShopServiceImpl extends ServiceImpl
→ ShopController
```

为什么按这个顺序：上层依赖下层的数据形状。Controller 在不知道 Service 方法和返回对象时没有稳定接口。

然后补充：

- `Result`：统一成功/失败 JSON。
- `WebExceptionAdvice`：捕获 RuntimeException。
- `MybatisConfig`：分页插件。

## 阶段三：先完成普通数据库业务

先做不带 Redis 的功能：

- 根据 ID 查询店铺。
- 新增和更新店铺。
- 分页查询店铺。
- 查询优惠券。
- 发布和查询笔记。

这样能先证明数据库表、实体映射和 HTTP 参数绑定是通的。如果一开始同时引入 Redis，出错时很难判断是数据库、序列化还是缓存问题。

## 阶段四：建立无 Session 的登录态

合理顺序：

```text
User / UserDTO / LoginFormDTO
→ 手机号校验
→ 验证码写 Redis
→ 登录校验
→ 首次登录自动注册
→ UUID token
→ UserDTO 转 Redis Hash
→ UserHolder ThreadLocal
→ RefreshTokenInterceptor
→ LoginInterceptor
→ MvcConfig 注册顺序
```

先有 token 存储，拦截器才知道应该从哪里恢复用户；先有 `UserHolder`，业务方法才可以取得当前用户。

## 阶段五：给热点查询加缓存

演进顺序：

```text
直接查 MySQL
→ Redis 命中直接返回
→ 未命中查库并回填
→ 空值缓存防穿透
→ 更新数据库后删缓存
→ 抽取 CacheClient
→ 互斥锁/逻辑过期备选
```

当前店铺查询实际启用的是空值缓存方案。互斥锁和逻辑过期虽在 `CacheClient` 中存在，但调用被注释，不要说成三种方案同时在线。

## 阶段六：优惠券与库存

先拆分两个数据概念：

- `tb_voucher`：通用券信息。
- `tb_seckill_voucher`：秒杀库存和起止时间。

新增秒杀券时：

1. 写 `tb_voucher`。
2. 写 `tb_seckill_voucher`。
3. 初始化 `seckill:stock:{voucherId}`。

这为后面的 Lua 秒杀准备了 Redis 库存。

## 阶段七：从同步下单演进到异步下单

应先理解最朴素的同步版本：

```text
检查时间和库存
→ 检查一人一单
→ MySQL 扣库存
→ 创建订单
```

它的问题是高并发请求全部进入数据库。然后逐步改造：

1. `RedisIdWorker` 生成趋势递增 ID。
2. Lua 原子判断 Redis 库存和一人一单。
3. Lua 预扣库存并写 Stream。
4. HTTP 请求快速返回 orderId。
5. 后台消费者读取 Stream。
6. 在 MySQL 事务中扣库存并保存订单。
7. 事务成功后 ACK。
8. 异常时处理 pending-list。

## 阶段八：补测试和失败路径

至少应该测试：

- Lua 返回 0/1/2 时接口结果。
- 返回给前端的 orderId 与消息中的 ID 一致。
- 重复消息不重复扣库存。
- DB 失败时不 ACK。
- pending-list 能恢复。
- 缓存不存在、空值、正常值和更新删除。
- token 为空、失效和有效三种拦截器路径。

当前真正有业务断言的测试主要是 orderId 一致性，其余多个测试只是占位烟雾测试。

## 面试回答“你怎么设计这个项目”

> 我先从数据库表和普通 CRUD 建立最小业务闭环，再加入 Redis 登录态和店铺缓存。秒杀部分没有一开始就上异步，而是先明确同步下单的一人一单和库存问题，再把入口资格校验收敛到 Lua，把订单写入 Redis Stream，由后台消费者事务落库。这样每一步都有明确要解决的问题。

## 自测

1. 为什么不应该先写 Controller 再考虑数据表？
2. 为什么先跑通 MySQL 再加 Redis 更容易调试？
3. `UserHolder` 为什么必须在需要当前用户的业务之前存在？
4. 秒杀为什么要先理解同步版本？
5. 当前测试覆盖最明显的空白是什么？
