# 05. 用户登录与双拦截器鉴权

## 本章目标

学完后要能从“发送验证码”一直讲到“请求结束清理 ThreadLocal”，并回答为什么要用两个拦截器。

## 业务目标

传统 Session 把登录状态存在单台服务器内存中，多实例部署会遇到 Session 共享问题。项目改为：

```text
客户端保存随机 token
Redis 保存 token → UserDTO
每次请求根据 token 恢复用户
```

## 从零编码顺序

```text
User / UserDTO / LoginFormDTO
→ Redis Key 常量
→ 手机号校验
→ 发送验证码
→ 登录与首次注册
→ token Hash
→ UserHolder
→ RefreshTokenInterceptor
→ loginInterceptor
→ MvcConfig
→ Axios authorization header
```

## 第一步：登录输入与安全输出

```java
public class LoginFormDTO {
    private String phone;
    private String code;
    private String password;
}
```

当前登录只使用 `phone` 和 `code`，`password` 是预留字段。

```java
public class UserDTO {
    private Long id;
    private String nickName;
    private String icon;
}
```

Redis 登录态只需要这些字段。完整 `User` 里的手机号和密码不应该进入客户端可见上下文。

## 第二步：手机号校验

```java
public static boolean isPhoneInvalid(String phone) {
    return mismatch(phone, RegexPatterns.PHONE_REGEX);
}

private static boolean mismatch(String str, String regex) {
    if (StrUtil.isBlank(str)) {
        return true;
    }
    return !str.matches(regex);
}
```

方法名是 `isPhoneInvalid`，返回 true 代表无效。阅读时不要把布尔语义看反。

## 第三步：发送验证码

Controller：

```java
@PostMapping("code")
public Result sendCode(@RequestParam("phone") String phone,
                       HttpSession session) {
    return userService.sendCode(phone, session);
}
```

虽然参数仍保留 `HttpSession`，Service 已经不使用 Session。

Service 核心：

```java
if (RegexUtils.isPhoneInvalid(phone)) {
    return Result.fail("手机号格式错误！");
}

String code = RandomUtil.randomNumbers(6);

stringRedisTemplate.opsForValue().set(
        LOGIN_CODE_KEY + phone,
        code,
        LOGIN_CODE_TTL,
        TimeUnit.HOURS
);
```

### Redis 数据

```text
Key:   login:code:13800138000
Type:  String
Value: 492831
TTL:   当前代码是 120 小时
```

这里有真实 bug：常量 `LOGIN_CODE_TTL = 120L` 看起来像 120 秒，但代码使用 `TimeUnit.HOURS`，最终是 120 小时。推荐改为明确的 2 分钟或 120 秒，并在常量名中体现单位。

最后：

```java
log.info("发送短信验证码成功，验证码：{}", code);
System.out.println("LOGIN_CODE phone=" + phone + " code=" + code);
return Result.ok();
```

演示环境用日志代替短信服务。生产环境不能把验证码输出到日志，否则任何能读取日志的人都可能登录用户账户。

## 第四步：验证码登录

```java
String phone = loginForm.getPhone();
if (RegexUtils.isPhoneInvalid(phone)) {
    return Result.fail("手机号格式错误！");
}

String cachecode = stringRedisTemplate.opsForValue()
        .get(LOGIN_CODE_KEY + phone);
String code = loginForm.getCode();

if (cachecode == null || !cachecode.equals(code)) {
    return Result.fail("验证码错误！");
}
```

逻辑是先做格式校验，再读取 Redis，避免无效手机号直接查询缓存。

当前缺口：登录成功后没有删除验证码，因此 TTL 内同一验证码可以重复使用。推荐成功后删除 `login:code:{phone}`。

## 第五步：首次登录自动注册

```java
User user = query().eq("phone", phone).one();

if (user == null) {
    user = createUserWithPhone(phone);
}
```

创建用户：

```java
public User createUserWithPhone(String phone) {
    User user = new User();
    user.setPhone(phone);
    user.setNickName(USER_NICK_NAME_PREFIX
            + RandomUtil.randomString(10));
    save(user);
    return user;
}
```

SQL 中手机号有唯一索引。若两个并发请求同时判断 `user == null`，数据库唯一索引会阻止真正重复账户，但代码没有把唯一键异常转换成“重新查询已有用户”，因此并发首次登录仍可能返回服务器异常。

## 第六步：生成 token

```java
String hashtoken = UUID.randomUUID().toString(true);
```

token 是随机 UUID，不包含用户 ID。客户端无法通过 token 本身推断用户身份。

变量名 `hashtoken` 不够准确，它只是 token；“Hash”指后续 Redis 数据类型，不是 token 做了哈希运算。

## 第七步：User 转为 Redis Hash

```java
UserDTO userDTO = BeanUtil.copyProperties(user, UserDTO.class);

Map<String, Object> userinfo = BeanUtil.beanToMap(
        userDTO,
        new HashMap<>(),
        CopyOptions.create()
                .setIgnoreNullValue(true)
                .setFieldValueEditor(
                        (fieldName, fieldValue) -> fieldValue.toString()
                )
);
```

为什么要转成 String：`StringRedisTemplate` 的 Hash key/value 序列化器面向字符串，`id` 原本是 Long，需要转为字符串。

写 Redis：

```java
String key = LOGIN_USER_KEY + hashtoken;
stringRedisTemplate.opsForHash().putAll(key, userinfo);
stringRedisTemplate.expire(key, LOGIN_USER_TTL, TimeUnit.MINUTES);
return Result.ok(hashtoken);
```

### Redis 数据

```text
Key: login:token:4f2c...
Type: Hash
Fields:
  id       → "42"
  nickName → "user_abcd..."
  icon     → ""
```

当前 `LOGIN_USER_TTL = 36000L` 且单位是分钟，即约 25 天。注释和数值没有清楚表达意图，应统一。

## 第八步：客户端携带 token

`common.js`：

```javascript
let token = sessionStorage.getItem("token");

axios.interceptors.request.use(config => {
  if (token) {
    config.headers['authorization'] = token;
  }
  return config;
});
```

- `sessionStorage` 在当前浏览器标签页会话中保存 token。
- 每次 Axios 请求自动加入 `authorization`。
- 项目没有使用标准 `Bearer <token>` 格式，而是直接传 UUID。

## 第九步：UserHolder

```java
public class UserHolder {
    private static final ThreadLocal<UserDTO> tl = new ThreadLocal<>();

    public static void saveUser(UserDTO user) {
        tl.set(user);
    }

    public static UserDTO getUser() {
        return tl.get();
    }

    public static void removeUser() {
        tl.remove();
    }
}
```

Tomcat 通常一个请求由一个工作线程处理。ThreadLocal 让同一线程上的 Controller/Service 不必层层传递 UserDTO。

线程池会复用线程，所以请求结束必须 `remove()`。否则下一个请求可能读到上一个用户，还会形成 ThreadLocal 内存泄漏风险。

## 第十步：刷新 token 的拦截器

### 读取请求头

```java
String token = request.getHeader("authorization");
if (StrUtil.isBlank(token)) {
    return true;
}
```

没有 token 仍然放行。这个拦截器只负责“尝试恢复用户”，不负责决定接口是否必须登录。

### 从 Redis 恢复用户

```java
String key = LOGIN_USER_KEY + token;
Map<Object, Object> userMap = stringRedisTemplate
        .opsForHash().entries(key);

if (userMap.isEmpty()) {
    return true;
}

UserDTO userDTO = BeanUtil.fillBeanWithMap(
        userMap, new UserDTO(), false
);
UserHolder.saveUser(userDTO);
```

如果 token 已过期，Hash 为空，也继续放行；后续是否被拦截交给第二个拦截器。

### 滑动续期

```java
stringRedisTemplate.expire(
        key, LOGIN_USER_TTL, TimeUnit.MINUTES
);
```

只要用户持续访问，token 的过期时间就持续往后延。这叫滑动过期。

### 请求结束清理

```java
@Override
public void afterCompletion(...) {
    UserHolder.removeUser();
}
```

即使 Controller 抛出异常，Spring MVC 完成请求处理后仍会执行清理。

## 第十一步：登录拦截器

```java
if (UserHolder.getUser() == null) {
    response.setStatus(401);
    return false;
}
return true;
```

它不再重复查询 Redis，只检查第一个拦截器是否恢复了用户。这就是职责拆分。

当前构造器仍接收 `StringRedisTemplate`，但实际逻辑完全没使用，这是可以清理的遗留依赖。

## 第十二步：注册顺序和放行路径

```java
registry.addInterceptor(
        new RefreshTokenInterceptor(stringRedisTemplate)
).addPathPatterns("/**").order(0);

registry.addInterceptor(
        new loginInterceptor(stringRedisTemplate)
).excludePathPatterns(
        "/user/code",
        "/user/login",
        "/shop/**",
        "/voucher/**",
        "/shop-type/**",
        "/upload/**",
        "/blog/hot",
        "/blog/**",
        "/seckill/**"
).order(1);
```

### 为什么 order 0 在前

如果鉴权拦截器先执行，ThreadLocal 还是空的，有效 token 也会被误判为未登录。

### 一个真实权限问题

`/blog/**` 被整体排除，意味着发布笔记、查询“我的笔记”和点赞都不会被 `loginInterceptor` 保护，但这些方法会调用 `UserHolder.getUser()` 或修改数据。公开查询只需要 `/blog/hot`，不应排除整个 `/blog/**`。

`/upload/**` 也完全公开，存在任意上传/删除风险。

秒杀真实路径是 `/voucher-order/seckill/{id}`，它没有被排除，因此当前仍需要登录。排除列表里的 `/seckill/**` 实际匹配不到这个入口。

## 登录、未登录、失效 token 三条路径

### 没有 token，访问公开接口

```text
RefreshToken: 放行
LoginInterceptor: 路径被排除
Controller: 正常执行
```

### 没有 token，访问受保护接口

```text
RefreshToken: 放行
LoginInterceptor: UserHolder 为空 → 401
```

### 有效 token

```text
RefreshToken: Redis Hash → UserDTO → ThreadLocal → 刷 TTL
LoginInterceptor: ThreadLocal 非空 → 放行
afterCompletion: remove
```

## 登出为什么尚未完成

Controller 当前：

```java
@PostMapping("/logout")
public Result logout() {
    return Result.fail("功能未完成");
}
```

推荐实现应读取 token 并删除 `login:token:{token}`。前端只清 sessionStorage 不足以让服务端 token 立即失效。

## 面试 30 秒讲法

> 登录成功后我生成随机 UUID token，把精简 UserDTO 存进 Redis Hash。请求进入时第一个拦截器负责从 Redis 恢复用户到 ThreadLocal 并刷新 TTL，第二个拦截器只判断受保护接口是否存在用户。请求结束在 afterCompletion 清理 ThreadLocal，防止线程池复用导致用户串号。

## 面试 2 分钟讲法

> 验证码先按手机号写入 Redis String，登录时校验验证码，首次登录自动注册。成功后生成不包含用户信息的 UUID token，将 User 转成只含 ID、昵称和头像的 UserDTO，再转为字符串 Map 存入 Redis Hash。前端每次在 authorization 头携带 token。全路径的 RefreshTokenInterceptor 尝试恢复用户并做滑动续期，后面的 LoginInterceptor 只对受保护接口判断 ThreadLocal。这样公开接口上的活跃用户也能续期，同时避免每个业务方法重复查询 Redis。当前代码的验证码 TTL 单位、日志泄露、过宽放行路径和登出未实现都是我会修正的点。

## 高频追问

### 为什么不用 Session？

Redis token 更方便多实例共享和统一失效，但仍需考虑 Redis 可用性、token 安全和过期策略。

### 为什么用 Hash？

UserDTO 是多个字段，Hash 可以按字段存储，也不需要把完整 JSON 每次整体覆盖。这个项目每次仍读取整个 Hash，优势主要是结构清晰。

### 为什么不能把 User 放 ThreadLocal 后不清理？

Web 容器复用线程，未清理会让后续请求读取旧用户，并持有对象引用。

### JWT 会不会更好？

JWT 可减少服务端会话查询，但立即注销、权限变更和续期更复杂。Redis token 需要一次缓存查询，却便于主动删除和集中管理。
